# テキストデータとファイル管理のコツ

## 1. テキストデータの容量について
文字（テキスト）だけであれば、5000文字でも1万文字でも、**容量は気にせずバンバンGitHubに保存して全く問題ありません！**

*   **文字データ（Markdownなど）の重さ**
    *   全角1文字 ＝ 2〜3 バイト
    *   **5,000文字の記事 ＝ 約 15 KB**
*   **他のデータの重さとの比較（目安）**
    *   スマホで撮った写真1枚 ＝ 約 3,000 KB（記事200本分！）
    *   短い音声データ ＝ 約 10,000 KB（記事600本分！）
    *   10分の動画 ＝ 約 100,000 KB（記事6,600本分！）

つまり、5,000文字の記事を1,000本書いたとしても、写真数枚〜十数枚分の容量（15MB程度）にしかなりません。テキストデータだけで容量がいっぱいになることはほぼ不可能です。

## 2. GitHubに貯めていく時のファイル管理のコツ
長文の記事や大量のプロンプトを貯めていく場合、金庫（リポジトリ）のなかに以下のように、「フォルダ（ディレクトリ）」を作って整理していくと便利です。

**【例：〇〇株式会社のメディア記事作成プロジェクトの場合】**
```
📁 client_A_articles (GitHubの金庫)
 ┣ 📁 prompts (プロンプトを保存するフォルダ)
 ┃ ┣ 📝 01_article_prompt.md (記事作成用プロンプト)
 ┃ ┗ 📝 02_rewriting_prompt.md (リライト用プロンプト)
 ┣ 📁 drafts (下書きを保存するフォルダ)
 ┃ ┣ 📝 20260301_draft_about_AI.md (5000文字の下書き記事)
 ┃ ┗ 📝 20260305_draft_marketing.md
 ┗ 📝 README.md (このプロジェクトの説明書)
```

**【管理のポイント】**
*   **プロンプト（指示書）と記事（結果）を別々にする**
    エディタ（Antigravity）上で新しくフォルダを作り、その中に「プロンプト（指示書）」と「生成された記事（出力結果）」をそれぞれ別々のMarkdownファイル（`.md`）としてどんどん保存していくと、過去の作業が把握しやすくなります。
*   **ファイル名には「英語」と「日付」や「用途」を入れる**
    日本語の名前はエラーの原因になることがあるため、半角英数字で `20260301_draft.md` のように日付や `_prompt`（指示書）と付けておくと、検索しやすくなります。

★ 「この記事の1週間前のバージョンに戻したいな」という時にも、GitHubなら一瞬で戻せるので、長文執筆には最高のツールになります！
